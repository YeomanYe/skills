# shadcn-admin: Domain Classification Examples

这个案例展示真实规则应如何按职责归入不同规则域。

## architecture

属于 `architecture` 的典型规则：

- 路由注册放在哪里
- feature 目录怎么拆
- URL 层级和目录层级怎么对应
- 页面私有组件、feature 组件、全局组件各自放在哪
- 模块结构如何与菜单、权限或业务域对齐

真实例子：

- “路由锚点和 `_authenticated` 集中注册”属于结构与落位，不属于编码风格
- “组件归层边界”属于代码放在哪，不属于视觉规范
- “`components`、`components/ui`、`hooks`、`lib` 属于通用资产层，应默认保持中性”属于结构边界
- “组件只正式分 `primitive` 和 `composite`”属于资产层组件边界定义

## coding

属于 `coding` 的典型规则：

- 命名风格
- 注释要求
- import 规则
- 数据流、service/store 边界
- i18n 写法
- 交付前必须做哪些验证

真实例子：

- “enum 成员必须有注释”属于编码约束
- “复杂计算不要直接塞进 render”属于编码风格
- “文案不得硬编码，必须走 i18n”属于编码规则
- “通用资产层文件的注释要求高于普通业务文件”属于编码规则
- “容器类 composite 组件可选使用 ASCII 结构图注释”属于编码注释规则
- “`coding/rules.md` 只保留原则和顶层约束”属于编码域结构设计
- “命令、验证、`.tmp/` 之类规则不应继续堆在编码总纲里”属于编码域内容归属规则

## completion

虽然“完成要求”通常仍放在 `coding` 域下，但它的职责应单独识别：

- 收尾检查
- 验证命令
- 清理要求
- 截图、录屏、验收证据

真实例子：

- “`.tmp/` 只能放临时产物”更适合收进完成/清理规则，而不是编码总纲
- “`lint/build` 必须执行并记录”更适合收进完成要求，而不是编码总纲

常见误放：

- 把所有零散规则都塞进 `completion`
  - 错因：实现期规则和收尾检查项不是一类职责

## ui

属于 `ui` 的典型规则：

- 视觉语言
- 布局骨架
- 组件视觉表现
- 页面模式
- 状态层级与交互反馈

真实例子：

- “中性、信息优先”属于视觉收敛原则
- “表格、卡片、表单的视觉层级怎么组织”属于 UI 规则
- “Button / Card / Input 的视觉边界”属于 UI 规则

## ai-guide

属于 `ai-guide` 的典型规则：

- AI 先读哪个入口
- 哪类任务应该继续读哪些规则
- AI 什么时候进入交付闸门
- AI 如何采集截图、录屏、验证证据
- AI 的输出格式和回流链路

真实例子：

- “AI 进入某个领域时先读该目录下的 `index.md`”属于 AI 消费方式
- “交付前是否需要 Playwright 录屏”属于 AI 执行与采证流程

## 常见误放示例

- 把“联调时 AI 应记录端口”放进 `architecture`
  - 错因：这是 AI 行为约束，不是项目结构事实
- 把“组件应该放到 `components/ui` 还是 feature 内”写进 `ui`
  - 错因：这是代码落位边界，不是视觉风格
- 把“入口文件里写满编码细则”当作 `CONTRIBUTING`
  - 错因：入口文件应负责路由，不应承载大量正文
