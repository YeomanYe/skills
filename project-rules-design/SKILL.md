# 项目规范架构

## 作用

这个 skill 用于处理“一个工程的规范体系应该怎么设计和怎么重构”。

它关注的不是单篇文档写法，而是整套规范系统：

- 应该包含哪些规则领域
- 每个领域的职责边界是什么
- 哪些内容放错层了
- 入口文件、索引文件和正文文件应该怎么分工
- 现有混乱规则应该怎么重构，而不是机械改名搬家

默认输出应是“规则架构方案”或“规则重构结果”，而不是只给零散建议。

## 参考案例

这个 skill 自带真实案例参考，优先按需读取，不要把当前项目整套文档直接当作模板照搬。

- `references/shadcn-admin-before-after.md`
  - 展示一个项目如何从 `RULE.md + ai/*.md` 重构为 `CONTRIBUTING.md + docs/<domain>/index.md + 子规则`
- `references/shadcn-admin-domain-classification.md`
  - 展示真实规则如何判断应归入 `architecture`、`coding`、`ui` 或 `ai-guide`
- `references/shadcn-admin-target-structure.md`
  - 展示一套已经落地的目标结构，以及各入口和二级文件的职责

使用原则：

- 参考的是“分层方式、职责边界、重构方法”，不是逐字复制文件内容
- 若当前项目的规则域明显不同，应沿用本 skill 的分类方法，而不是强行套用该案例文件名
- 只有在用户明确要求“参考真实案例”或“给一个落地样板”时，再继续读取这些案例

## 适用时机

以下情况使用本 skill：

- 用户想新建一个工程规范体系
- 用户想整理已有的项目规范、规则文档或 AI 协作规则
- 用户在讨论 `CONTRIBUTING.md`、`docs/<domain>/index.md`、规则入口、规则路由、规范分层
- 用户想判断某条规则属于 architecture、coding、ui 还是 ai-guide
- 用户想把散乱规则重构为更稳定的目录结构和文档体系

以下情况通常不要使用：

- 只写单个功能设计文档
- 只修正文案或格式
- 只讨论某个语言或框架的局部编码建议
- 只改单个 feature 的业务说明文档

## 核心原则

1. 先做规则盘点，再做结构设计
2. 先按职责分类，再决定文件名和目录
3. 不做机械映射，必须按内容重构
4. 入口文件只负责路由，不负责堆正文
5. 一个领域只定义一种职责，不把不同层的规则混在一起
6. 任何重构都应避免并存两套“同时有效”的规则体系

## 默认规则模型

若项目没有更合适的特殊需求，优先从以下模型开始：

- `CONTRIBUTING.md`
  - 唯一总入口
  - 定义阅读顺序、任务路由、优先级
- `architecture`
  - 代码写在哪
  - 模块怎么拆
  - 层级怎么对应
  - 目录和落位边界
- `coding`
  - 代码怎么写
  - 命名、注释、数据流、错误处理、交付要求
- `ui`
  - 视觉语言
  - 布局骨架
  - 组件视觉规则
  - 页面模式
- `ai-guide`
  - AI 先读什么
  - AI 何时继续下钻
  - AI 如何执行、采证、回流

在该默认模型下，若项目存在明确的通用资产层，应额外识别并单独处理：

- `components`
- `components/ui`
- `hooks`
- `lib`

这些目录通常不应被当作普通业务目录看待，而应按“通用资产层”判断：

- 默认保持中性
- 默认不轻易长出业务属性
- 一旦进入这些目录，注释和边界说明要求应高于普通业务文件

当 `coding` 域中存在类似 `rules.md` 的总纲文件时，应额外保持这一层的纯度：

- 总纲文件优先写“编码原则”和少量跨编码域顶层约束
- 不要把命令、验证步骤、`.tmp/` 规则、shadcn 选择、专题数据流约束等杂项继续堆在总纲文件里
- 这些具体规则应下沉到更贴近职责的二级文件，例如 `data-flow`、`completion`、`comments`、`ui/components`

只有当项目确实需要时，再增加额外领域，例如：

- `testing`
- `security`
- `ops`
- `api`
- `data`

## 必要流程

### 1. 盘点现状

先读当前项目已有的规则入口和正文，例如：

- `CONTRIBUTING.md`
- `RULE.md`
- `docs/`
- `ai/`
- `AGENTS.md`
- skill 相关规则入口

盘点时至少要回答：

- 当前有哪些规则文档
- 哪些是入口，哪些是正文
- 哪些内容重复
- 哪些内容放错层
- 哪些领域缺失

### 2. 建立分类模型

把现有规则按职责分类，而不是按旧文件名分类。

至少应判断：

- 这是结构规则还是编码规则
- 这是视觉规则还是 AI 行为规则
- 这是入口说明还是正文约束
- 这是项目本体规则还是 AI 消费方式
- 这是否属于通用资产层的边界规则
- 这是否是在约束资产层文件不要被业务污染

如果项目存在通用资产层，还应继续判断：

- 哪些目录属于通用资产层
- 这些目录是否被错误地允许承载业务文案、业务路由、业务接口、页面专属行为
- 对 `components` 的分类是否足够清晰
- 对 `hooks` 和 `lib` 的注释要求是否足够明确

如果项目已经存在 `coding/rules.md` 或等价总纲文件，还应继续判断：

- 其中的 `MUST` / `SHOULD` 是不是在展开总纲原则，而不是零散清单
- 命令、验证步骤、交付检查项是否被误放进总纲
- 某些小规则是应该下沉到 `completion`、`data-flow`、`naming`、`comments` 还是 `ui` 域

### 3. 识别问题

重点识别以下问题：

- 入口文件夹带过多正文
- 正文里混入其它领域的规则
- 同一规则在多处重复
- 文件名或目录名不能表达职责
- 同时存在两套有效规则体系
- AI 无法判断先读什么、再读什么
- 通用资产层与业务层边界不清
- `components`、`hooks`、`lib` 被写成“看似通用，实则带业务属性”
- 资产层文件缺少足够清晰的功能、参数、适用边界说明
- `coding` 总纲文件夹杂过多专题小规则，失去总纲作用
- `MUST` / `SHOULD` 只是零散清单，没有真正承接总纲原则
- 命令、交付项、验证项被误塞到编码总纲中

### 4. 设计目标结构

目标结构至少应说明：

- 总入口文件是什么
- 分几个领域
- 每个领域是否需要 `index.md`
- 每个领域下有哪些二级文件
- 各文件之间的优先级

若项目有较强的前端资产层，还应明确：

- 是否单独定义通用资产层边界
- `components`、`components/ui`、`hooks`、`lib` 是否被视为更高稳定性的目录
- 组件在通用资产层中如何分类
- 资产层文件需要什么级别的注释

若 `coding` 域采用“总纲 + 细则”结构，还应明确：

- 哪个文件是编码总纲
- 哪些约束属于总纲级
- 哪些约束应下沉到专题细则
- `completion` 只放收尾检查项，不接收所有实现期小规则

如果用户只是想分析，应输出结构方案。
如果用户明确要求落地，则继续创建或修改文件。

若用户明确要求参考实例，或当前项目的规则明显混乱、难以仅凭抽象原则判断，则应读取最相关的案例文件：

- 需要看重构思路时，读 `references/shadcn-admin-before-after.md`
- 需要判断某条规则放哪一层时，读 `references/shadcn-admin-domain-classification.md`
- 需要给出目标目录骨架时，读 `references/shadcn-admin-target-structure.md`

### 5. 落地重构

如果用户要求直接实施：

- 创建新的入口文件和分域目录
- 按职责迁移规则内容
- 更新活跃引用路径
- 删除或停用旧入口，避免双轨并存

不要只改文件名而不重写内容边界。

### 6. 输出结果

最终至少应输出：

- 当前问题摘要
- 新的规则模型
- 文档结构
- 哪些内容被移动到了哪里
- 哪些内容被删除或合并
- 仍需后续处理的历史引用或旧文档

## 结构建议

若使用目录化规则，优先使用：

```text
CONTRIBUTING.md

docs/
  architecture/
    index.md
    ...
  coding/
    index.md
    ...
  ui/
    index.md
    ...
  ai-guide/
    index.md
    ...
```

规则：

- `CONTRIBUTING.md` 只做总入口
- 每个领域目录的 `index.md` 只做该领域入口
- 具体规则写在二级文件
- 不让 `index.md` 重新长成大杂烩正文

若项目是前端或全栈 UI 项目，优先考虑在 `architecture` 或 `coding` 中显式定义以下内容：

- 通用资产层目录边界
- `primitive` / `composite` 组件分类
- 资产层组件、hooks、lib 的注释要求
- 何时允许 ASCII 结构图作为容器组件的辅助注释

若项目采用 `coding/rules.md` 之类的总纲文件，还优先考虑显式定义：

- 总纲文件只保留原则和顶层约束
- 小规则应按职责下沉
- `completion` 只承担项目级完成与收尾检查

## 禁止行为

- 只做文件重命名，不重构内容归属
- 把 architecture、coding、ui、ai-guide 混成一层
- 让入口文件承载大量具体规则正文
- 保留两套同时有效的旧新规则体系
- 只给抽象建议，不输出结构方案或落地结果

## 输出契约

默认输出结构：

```md
## 规范现状

## 规则分类模型

## 建议结构

## 迁移与重构建议

## 若直接落地，应修改的文件
```

若已实际重构，则额外输出：

- 已创建 / 已修改文件
- 已删除旧入口或旧目录
- 仍未处理的历史引用范围
