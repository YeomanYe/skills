# Web Image

## Overview

用于把“网页出图”任务落成可交付位图。先判断是否命中特殊场景；命中则走对应约束分支，否则走通用 HTML/CSS 出图流程。

默认目标不是“做一张看起来还行的图”，而是输出：

- 固定尺寸的成品位图
- 可复用的 HTML/CSS 源文件
- 已经过尺寸与溢出检查的结果

## Trigger

命中以下任一场景应使用本 skill：

- 用户要求生成或修改固定尺寸的网页宣传图、海报、横幅、封面图、OG image、社媒图
- 用户要求用 HTML/CSS 排版并导出位图
- 用户要求把真实截图、图标、标题文案排成营销图
- 用户要求商店素材图，例如扩展商店图、promo tile、marquee、feature graphic
- 用户已有一张由 HTML/CSS 生成的图片，要求继续微调版式、字距、层级、留白、装饰、截图使用

## Do Not Use

以下情况不要使用本 skill：

- 用户明确要文生图、照片感图像、插画、材质生成
  这种情况应交给图像生成能力
- 任务本质是修改 SVG、logo 系统、图标集
- 任务本质是改页面 UI，而不是导出一张位图
- 没有固定尺寸要求，也没有明确出图目标

## Workflow

### 1. 先判定是否是特殊场景

优先判定以下特殊场景：

- 有平台硬尺寸或槽位要求
  例如 `440x280`、`1400x560`、`1280x800`、`1200x630`
- 必须使用真实截图
- 必须复用项目 icon、品牌色、现有 UI 片段
- 有明确的信息层级上限
  例如“小图只允许一句主文案”或“两排标题”
- 有安全区要求
  例如底部不能贴边、标题不能溢出、元素不得被裁切

命中特殊场景时，不要直接套通用模板；先锁死该场景的约束，再开始做图。

### 2. 特殊场景优先

常见特殊分支：

- 扩展商店素材
  要先按平台尺寸、截图规则、宣传图规则处理
- 截图说明图
  必须让真实截图保持主体可信，不要用装饰抢主视觉
- 小尺寸宣传图
  优先级是主视觉和一句能读懂的卖点，不是堆信息
- 继续迭代已有出图
  优先在现有模板上改，避免重做导致风格断裂

### 3. 若不是特殊场景，走通用流程

通用流程：

1. 锁画布尺寸
2. 定主视觉
3. 定文字层级
4. 只保留必要元素
5. 用 HTML/CSS 落盘
6. 导出位图
7. 检查尺寸、裁切、溢出、可读性

## Hard Rules

### 固定画布优先

- 出图前必须先锁定目标尺寸
- 不允许边做边猜尺寸
- 小图和大图不能复用同一信息密度

### 信息层级优先于信息总量

- 先决定“用户第一眼应该看到什么”
- 小图优先单主视觉 + 极短文案
- 如果主标题已经承担说明责任，删除多余辅助文案
- 不要因为有空间就继续加标签、说明、徽章

### 真实素材约束

- 如果任务要求使用真实截图，必须使用真实截图
- 不要用抽象装饰替代截图的说明职责
- 如果任务要求直接使用现有 icon，就不要再对 icon 做二次包装导致主体变形或被裁切

### 安全区与裁切

- 标题、截图主体、底部信息块都必须留安全区
- 任何底部卡片、说明、proof 行都不得贴边或被截断
- 图标不得被画布裁切
- 长文案必须先通过字号、字距、断行策略解决，不能硬挤

### 装饰元素

- 装饰必须服务主题语言，不能只为了“占空间”
- 装饰不能抢主视觉
- 如果装饰和主题不一致，优先删除或重做，不做小修小补

## HTML/CSS Implementation Guidance

- 默认使用 HTML/CSS 渲染导出
- 优先复用已有模板；仅在方向错误时整体重构
- 对单个槽位的特殊需求，优先增加场景开关，而不是把通用模板改坏
- 场景差异应显式编码
  例如 `hideBadges`、`titleStyle`、`extraBottomSpace`
- 修改已有成品时，优先做局部可控改动，再导出验证

## Verification

交付前至少检查：

- 实际像素尺寸正确
- 没有文字溢出或被裁切
- 没有图标或截图主体被裁切
- 底部区块完整显示
- 主标题可在一眼内读懂
- 小图没有出现“大图式信息密度”

如果用户反复要求微调，优先把问题归类为：

- 信息层级错误
- 断行策略错误
- 间距节奏错误
- 主次关系错误
- 装饰语言错误

不要每轮都无方向地挪元素。

## Deliverables

完成时应真实产出：

- 导出图片文件
- 对应 HTML/CSS 源文件
- 简短说明本轮改动集中在哪个维度
  例如标题层级、留白、安全区、装饰语言、截图使用

## Examples

历史案例存放在 `examples/`。开始新任务前可先 `ls examples/` 找尺寸、场景或品牌调性接近的 case，复用骨架与节奏，但**不要把案例的具体规则抽象成全局原则写进本文档**——案例之间存在冲突，每个 case 只在它自己的上下文内自洽。

每个案例目录至少包含：

- `README.md` — 必须显式分节列出：上下文、关键决策、**可复用的部分**、**需要按项目改的部分**。缺少"可复用 / 需要改"两个分节的案例视为无效，不应被其他任务参考
- 源码（`*.html` / `*.css`）
- 成品位图

当案例规则与上游 skill（例如 `flow-ext-publish`）的硬规则冲突时，遵从上游硬规则。案例只在它自己的上下文内自洽，不得凌驾于链路上已声明的约束。
